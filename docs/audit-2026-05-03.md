Read all 9 AGENTS.md files, the three guidelines, the 2026-05-01 audit, mapped the codebase, and ran three parallel deep-dives (web / api / packages) with personal verification of the highest-leverage claims. Findings are grouped by surface and severity, with file:line citations. ## Critical (security / cross-tenant / shipping-blocking)

### C1. SES webhook secret is **optional** — unauthenticated webhook in any env without `SES_WEBHOOK_SECRET`

`apps/api/src/webhooks/ses.ts:22-25` — `if (secret && c.req.header(...))`. If env is unset (dev, mis-configured prod), **anyone** can POST forged SNS notifications, write `webhook_event` rows for arbitrary `provider_message_id`, and trigger `inngest.send` with cross-workspace effects. The query at line 40 finds the workspace by provider message ID, so an attacker who guesses an outbound ID corrupts a real workspace's bounce/complaint state.

Fix: require the secret unconditionally; in production, also verify SNS signature against AWS public cert.

### C2. File presign trusts client-supplied `mimeType` for downstream `Content-Type`

`apps/api/src/files.ts:101-130`. Client picks `mimeType`, server validates against allowlist, but the presigned PUT embeds that `ContentType` only as a *signed expectation* — the bucket policy doesn't enforce it. Even if it did, the `Content-Disposition` is unset, so `image/svg+xml` (which IS in the allowlist) is a stored XSS vector when served via signed GET. Note SVG is allowlisted alongside `application/zip`.

Fix: drop SVG from `ALLOWED_MIME` (or strip on upload via SVGO/sanitize), set `Content-Disposition: attachment` on signed GETs, and add an S3 bucket policy condition requiring `Content-Type` to match the signed value.

### C3. `BrandSplash` wired as `pendingComponent` of the `/app` route

`apps/web/src/routes/app.tsx:35`. Per `frontend.md` §14 + anti-pattern #32, this turns every internal navigation that misses cache into a full-screen brand splash — the exact "instant downgrade from feels-like-Linear" trap the guideline names. The session is module-cached (`session-loader.ts`) so this is rare in practice, but anything that ever makes `beforeLoad` slow (Phase 6 SSR, a new prefetch) re-triggers it. **Regression** since the audit.

Fix: replace with `RoutePendingFeedback` (or omit and rely on the `index.html` inline splash for the cold-start window).

### C4. `customerList` cap is `ALL_TICKET_MESSAGE_LIMIT` (2000), not `MAX_LIST_LIMIT` (1000)

`packages/zero-schema/src/queries.ts:232` — uses the wrong constant, allowing 2× the documented display ceiling. Pagination sentinel logic and frontend memory budget assume 1000.

### C5. Six unbounded list queries (no `.limit()`)

`packages/zero-schema/src/queries.ts:479-516` — `tagGroups`, `tagGroupsForSettings`, `tags`, `tagsForSettings`, `customFieldsByCategory`, `customFieldsForSettings`. A workspace with 5k tags syncs all 5k rows on first paint and keeps a subscription open. Violates `frontend.md` §18 #29 explicitly.

### C6. `route-inbound-message.ts` is a 1503-line monolith

Maintainability ceiling exceeded ~3×. Single file mixes parse, threading, customer ID resolution, routing rules, rate limit, persistence, audit. Phase 4 (chat / SMS) cannot reuse anything here. This is the single biggest tech-debt liability in the repo.

## High (correctness / a11y / data loss)

### H1. `window.confirm('Delete this note?')` in note deletion

`apps/web/src/components/customer/note-card.tsx:57` — violates `copy-guide.md` §3 (verdict-style modal) and breaks the design language (browser-native dialog).

### H2. Note composer has no draft persistence (still open from 2026-05-01 audit)

`apps/web/src/components/customer/note-composer.tsx:35` — plain `useState(initialBody ?? '')`. Main `composer.tsx` persists to Zustand+localStorage. Customers/tickets lose half-typed notes on accidental nav. **Still open.**

### H3. `outline-none` without `focus-visible:ring-*` in the command palette

`apps/web/src/components/command/command-palette.tsx:290, 360, 389, 430` (5 sites). New surface from Phase 30; keyboard-only and screen-reader users have no focus indication. Violates `frontend.md` §18 #4 and the `:focus-visible` rule in §15.

### H4. `inboxOpen` is preloaded with `MAX_INBOX_LIMIT` (2000) on every customer-timeline render

`apps/web/src/components/timeline/timeline-feed.tsx:148` — `useQuery(queries.inboxOpen({ limit: MAX_INBOX_LIMIT }), CACHE_FOREVER)` inside the customer-scoped timeline. The customer view doesn't render inbox tickets; this subscription is dead weight that doubles steady-state bandwidth on the customer profile.

### H5. Inbound email body has no size cap before in-memory parse

`apps/api/src/inngest/functions/route-inbound-message.ts` (load-and-parse path around lines 308-320 per the api-audit). `simpleParser` runs on the full byte array; a 100 MB SES delivery OOMs the worker. Add a 25–50 MB hard cap on `GetObjectCommand` and stream-to-disk parser.

### H6. Routing-rule regex evaluator has ReDoS surface

`route-inbound-message.ts:1435-1442` — fallback path interpolates user pattern into `^${escaped}$` after escaping. The original `new RegExp(pattern, 'i')` path runs *unbounded* user regex against email subjects/headers. Pattern `(a+)+b` against a long no-match string hangs the Node thread. Add a regex timeout (`vm.runInNewContext` with a budget, or `safe-regex2` rejection) and run on every routing rule write.

### H7. Inbound auth-results (SPF/DKIM/DMARC) are extracted and logged but never gate decisions

`route-inbound-message.ts:738-740`. Auto-responder detection is independent of DMARC verdict. Spoofed `auto-submitted: no` from `dmarc=fail` mail still routes as a customer message. AGENTS.md root §"Email subsystem" calls this out as a planned concern; not yet enforced.

### H8. Stale-effect race on `setActiveTabTitle` (still open from frontend.md §43)

`apps/web/src/components/timeline/timeline-feed.tsx:217-226, 475-482` — sets workbench tab title without `expectedRouteId` guard. Async router nav + Zero re-emit = wrong tab gets stamped. The pattern was documented but never adopted.

### H9. `useEffect` for derived state in custom-fields-block

`apps/web/src/components/conversation/custom-fields-block.tsx` — multiple effects mirror prop → local state and re-run autoresize. Should be `useMemo` + ref-callback. Anti-pattern #10.

### H10. Customer email lookup is case-sensitive

Inbound email lookup compares emails verbatim; `Alice@example.com` and `alice@example.com` create two customer rows. Normalize to lowercase pre-lookup and pre-insert.

### H11. Customer `addAlternateEmail` swallows UNIQUE-violation

`route-inbound-message.ts:1186` — `.catch(() => null)`. Hides duplicate-key errors and any other DB error. Either pre-check existence or branch on `error.code === '23505'` only.

## Medium (drift / inconsistency)

### M1. `ticket.update` mutator redundant cast

`packages/mutators/src/index.ts:223` — `as WorkspaceAuthData` after `assertCanModifyTicket`. The assertion already narrows the type. Cosmetic, but inconsistent with `ticket.assign` (line 274).

### M2. `customerList` orderBy lacks PK tiebreaker

`packages/zero-schema/src/queries.ts:250-253` — three orderBy keys, none of them PK. Equal `lastSeenAt` + `updatedAt` → non-deterministic row order → j/k jumps unpredictably. `frontend.md` §3.2 makes the tiebreaker mandatory.

### M3. Hardcoded limits inside queries

`queries.ts:364` (`Math.min(... ?? 10, 200)`), `queries.ts:425` (`200`). Should use `CUSTOMER_TICKET_LIMIT` / `DEFAULT_RELATED_TICKET_LIMIT` from `consts.ts`.

### M4. `updateInboundRaw` builds SQL `SET` clause via `sql.unsafe(...)`

`apps/api/src/inngest/functions/route-inbound-message.ts:871` — **not currently injectable** (only static column names go into `sets[]`, user data flows through `$N` params). Downgraded from "critical" to "fragile". Replace with Drizzle `update().set({...})` which is ~5 lines shorter and impossible to mis-extend.

### M5. SES SNS auto-confirm `await fetch(SubscribeURL)` has no try/catch

`webhooks/ses.ts:32`. If the fetch throws, request returns 500 yet SNS may still confirm. Wrap and log.

### M6. `dev` inbound email endpoint is on regardless of `NODE_ENV`

`apps/api/src/inbound/email.ts:165` (`handleDevInboundEmail`). Should refuse with 404 in production.

### M7. Pagination patterns: still two (no longer three)

`limit + 1` sentinel everywhere except inbox's growing window, which is documented as an exception in `consts.ts:8-10`. Audit-listed item is now an *intentional* divergence — keep it, but call it out in `conventions.md` §1 (it currently shows the pattern but doesn't enumerate the exception explicitly enough).

### M8. Empty-state copy drift ("yet")

`settings.channels.email.index.tsx:62, 73, 82` — "No domains yet", "No addresses yet", "No rules yet". `copy-guide.md` §2 says drop "yet" when not temporal. **Still open from 2026-05-01.**

### M9. Drizzle ↔ Zero parity is hand-mirrored with no CI check

`packages/db/src/schema/*` vs `packages/zero-schema/src/schema.ts`. Adding a column to Drizzle, forgetting to mirror it in Zero, or extending a `pgEnum` without bumping the `enumeration<>()` union compiles cleanly and fails at runtime. AGENTS.md root §"Postgres + Drizzle" says "Plan a CI drift check" — still not planned.

### M10. `apps/web/src/components/conversation/custom-fields-block.tsx` is 567 lines

Approaching the 500-line ceiling. Split editor row into its own component before it crosses.

### M11. Recovery cron idempotency keys not verified

`server-mutators.ts:84` uses `msg-req-${args.id}` for the live send. The recovery path needs the same key (or a deterministic derivative) to avoid re-delivery. Audit; if unverified, add an integration test.

### M12. Magic markers for threading (`::tid:<id>::`) are predictable

A reply that contains those tokens (test emails, support escalations) will be misthreaded. Replace with a per-workspace HMAC prefix.

## Patterns that are demonstrably solid (don't regress)

- `applyWorkspaceScope` is consistently used; no `.where('workspaceID', ...)` outside it.
- `crypto.randomUUID()` everywhere; no `nanoid` for UUID columns.
- `MutationError` + `MutationErrorCode` enum is consistent in mutators.
- HMAC reply-token uses `timingSafeEqual` correctly (`apps/api/src/email/reply-token.ts:117-119`).
- Two-pass auth middleware works as documented.
- Inngest event names avoid "email.*" — uses `delivery/message.*` per the polymorphic-channel commitment.
- Session module-cache (`session-loader.ts`) is in place and prevents `beforeLoad` thrash.
- Magic numbers were extracted to `consts.ts` and `lib/list-constants.ts` — that fix from the audit landed.
- `noUncheckedIndexedAccess: true` is on; the codebase has almost no `!.` non-null asserts (good).

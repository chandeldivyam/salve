# Codebase Audit — 2026-05-01

A deep audit of `apps/web` against the zbugs reference implementation
(`/tmp/zero-mono/apps/zbugs`, by Rocicorp — creators of Zero) and against
mainstream React/SaaS best practices. Goal: identify drift, missing
abstractions, and copy inconsistencies, and propose a concrete refactor
plan that locks in a uniform feel as the product grows.

Companion documents produced alongside this audit:

- [`guidelines/conventions.md`](../guidelines/conventions.md) — engineering rules going forward.
- [`guidelines/copy-guide.md`](../guidelines/copy-guide.md) — UX writing / tone guide.

## 2. Patterns to adopt from zbugs

zbugs (the Rocicorp reference at `/tmp/zero-mono/apps/zbugs`) is intentionally
lean. Where we already mirror it, we should enforce. Where we drift, we
should converge.

### Query layer

- **One file, all queries.** zbugs centralises in `shared/queries.ts` (454
  lines). We already do this in `packages/zero-schema/src/queries.ts`
  (773 lines). ✅ Keep enforcing — no inline `defineQuery` in routes or
  components.
- **Filter composition uses Zero's `and`/`or` builder, not a custom
  abstraction.** zbugs deliberately keeps filters ad-hoc per query
  (`shared/queries.ts:404-435`). The Zero docs reinforce this: "Keep ZQL
  expressions simple; return direct ZQL rather than complex business logic
  in query definitions." Counter to typical FastAPI-style CRUD generalisation:
  in Zero, the builder *is* the abstraction. Don't add a layer.
- **Pagination = `limit + 1` sentinel for paginated lists.** Constants
  like `INITIAL_COMMENT_LIMIT` live in `shared/consts.ts`, not per-file.
  zbugs's preload uses absolute limits (no sentinel) — same as our inbox.
- **Cache policy = named constants** (`CACHE_NAV`, `CACHE_PRELOAD`). We
  already mirror this in `apps/web/src/lib/zero-cache.ts:1-26`. ✅ Enforce
  no inline `{ ttl: '5m' }` anywhere.
- **Always validate query args with Zod.** Per Zero docs: "parameters come
  from the client and are untrusted." Use `defineQuery` with a Zod schema.

### Mutators

- **Auth check before existence check.** zbugs `mutators.ts:104` does this
  to prevent information disclosure ("does ticket X exist?"). Worth
  auditing every opendesk mutator for the same ordering.
- **Typed error classes with codes** (`shared/error.ts`):
  `class MutationError<T extends MutationErrorCode>`. Codes are easier to
  match in UI than strings. Worth adopting.
- **Replicache/Zero error policy:** retry on temporary, increment
  `lastMutationID` and skip on permanent. Follow this in server-side
  mutator implementations to avoid client deadlock loops.

### Frontend

- **Lists use `@tanstack/react-virtual` directly** in zbugs
  (`useZeroVirtualizer` is a thin wrapper). No table framework — your data
  is row-shaped, you don't need column models or sorting plumbing.
- **TanStack Table** is the right choice only when you need column-level
  features (resizing, hiding, filtering, grouping) — *we don't*, today.
  Ship with `<DataList>` (custom row + react-virtual when needed).
- **Skeletons = a CSS shimmer class on a div.** No skeleton component
  library in zbugs. They have one CSS class `skeleton-shimmer`.
- **Keyboard shortcuts** = single hook (`use-keypress.ts`) that
  *explicitly guards* against textarea / input / contenteditable. We have
  `useShortcut` — worth verifying it does the same.
- **Inline copy, no i18n.** Don't build a string catalog yet. (Linear,
  Atlas do the same in their early SaaS form.)
- **No barrel files.** Imports go to leaf modules. We partly do this;
  enforce.

### Routing

- zbugs uses Wouter (lightweight). We use TanStack Router — fine, but adopt
  TanStack Router's stronger feature: **search params as state**. Per the
  TanStack blog post "Search Params Are State", filters / sort / pagination
  belong in the URL, validated at the route level with Zod via
  `validateSearch`. This gives us deep linking, shareable filtered views,
  and back-button correctness for free.

## 3. Drift in the current opendesk codebase

### Query patterns (medium drift)

Most queries route through `packages/zero-schema/src/queries.ts` with named
cache constants. Drift points:

- `inbox-list.tsx:115` uses `queries.inboxOpen({ limit: pageLimit })` —
  absolute limit, no `+1` sentinel for "has more" detection.
- `timeline-feed.tsx:139` hardcodes `limit: 2000` for inbox; lines 141, 144,
  148, 162, 166 hardcode `200`, `200`, `50`, `2000`, `2000`.
- `customers.index.tsx:33` defines `const PAGE = 50` locally rather than
  importing a shared constant.
- `inbox-list.tsx:52-54` defines `INITIAL_PAGE = 200`, `PAGE_GROWTH = 200`,
  `PAGE_CEILING = 2000` locally.

**Fix:** centralise in `packages/zero-schema/src/consts.ts` (mirroring
zbugs `shared/consts.ts`).

### List / table UI (high drift)

Five distinct list shells, none shared:

1. **Inbox list** (`inbox-list.tsx`, 525 lines) — virtualised infinite
   scroll, j/k, filter pills, search.
2. **Customers index** (`customers.index.tsx`, 331 lines) — static
   pagination via "Show more" button, sort menu, search, column headers.
3. **Tags settings** (`settings.tags.tsx`, 838 lines) — card grid, no
   pagination, archived toggle.
4. **Custom fields settings** (`settings.custom-fields.tsx`, 598 lines) —
   card grid, category toggle, archived section.
5. **Email addresses** (`settings.channels.email.addresses.tsx`) —
   domain-grouped cards, no pagination.

Three independent skeleton implementations: `InboxRowSkeleton`,
`InboxListSkeleton`, `TicketDetailSkeleton`, `CustomerListSkeleton` — all
hand-rolled.

Two pagination patterns: "Show more" (customers) and infinite scroll
(inbox). No shared `<PaginationControl>`.

Two empty-state styles: settings shares `<EmptyState>` ✅; inbox and
customers each roll their own.

**Fix:** Build a single `<DataList>` primitive (see §5).

### Composer (medium drift)

Two composers:

- `composer.tsx` (763 lines) — Tiptap rich editor, reply / note tabs,
  email-address picker, S3 upload, draft persistence (Zustand +
  localStorage), Cmd+Enter to send.
- `customer/note-composer.tsx` (224 lines) — plain `<textarea>`, no
  toolbar, no draft persistence.

Tiptap docs explicitly recommend draft persistence via JSON to localStorage
or backend. Our note composer drops drafts on close.

**Fix:** extract `<ComposerShell>` (toolbar slot, tabs, send button, draft
persistence) and two editors (`<TiptapEditor>`, `<PlainEditor>`). Note
composer reuses shell + plain editor; gets draft persistence for free.

### Page layout (medium drift)

Settings routes share `<SettingsHeader>` + `<SettingsBody>` ✅. Non-settings
routes (`customers.index.tsx`, inbox routes, ticket detail) each roll their
own header (title, search slot, action slot).

**Fix:** build a `<PageHeader>` peer of `<SettingsHeader>`. Slots: `title`,
`description`, `actions`, `search`.

### File size (high drift)

The mainstream consensus across React style guides puts the comfortable
ceiling at **200–300 lines** per component file, with **500 lines** as the
last-resort upper bound before splitting becomes urgent. Files over budget:

| File | Lines | Notes |
|------|-------|-------|
| `components/timeline/timeline-feed.tsx` | 1325 | Two views + 7 queries. Split into `single-ticket-view.tsx` + `customer-view.tsx`. |
| `routes/app/settings.tags.tsx` | 838 | Extract `TagsGrid`, `TagGroupEdit`, `TagCreate`. |
| `components/composer.tsx` | 763 | Extract `ComposerShell`, `TiptapEditor`, `PlainEditor`, `FromPicker`, `AttachmentPreview`. |
| `routes/app/settings.custom-fields.tsx` | 598 | Extract sub-components. |
| `lib/workbench/store.ts` | 569 | Zustand store with 50+ setters. |

### Code quality (low drift)

- **No `any`** in hand-written code (only in generated `routeTree.gen.ts`).
- **No `!.` non-null assertions** in hand-written code.
- **No TODO / FIXME** comments — clean discipline.
- Comment quality varies — some files are model-quality (`zero-cache.ts:1-15`,
  `inbox-row.tsx:1-7`), some timeline files lean over-commented.

### Copy (medium drift)

Empty-state voice drifts across pages:

| Page | Copy | Voice |
|------|------|-------|
| Inbox | "Your inbox is empty" + "Replies will appear here." | Reassuring |
| Customers | "No customers match this view yet." | Neutral |
| Tags | "No tags yet" + "Group tickets so agents can filter…" | Task-focused |
| Custom fields | "No fields yet" + "Define operational…" | Feature-focused |
| Email domains | "Add a domain first" | Directive |

Button labels mix "Add tag" / "Add domain" / "Add address" / "New group" —
should be one or the other.

Pagination affordance: "Show more" (customers) vs infinite (inbox) — pick
one default per shape.

**Fix:** see [`guidelines/copy-guide.md`](../guidelines/copy-guide.md).

## 4. The CRUD-generalisation question (your FastAPI parallel)

You asked whether we should build general CRUD primitives the way FastAPI
projects often do (one `paginate(model, filters, sort)` helper). Honest
split:

### Backend / Zero — *resist generic CRUD layers*

- Zero's query builder + mutators *are* the abstraction. Wrapping
  `customerList`, `inboxOpen`, etc. in a generic
  `paginate(table, filters, sort)`:
  - Loses Zero's IVM optimisation.
  - Makes types harder.
  - Saves ~5–10 lines per query — not enough to justify the layer.
- Zero docs explicitly recommend keeping query implementations simple and
  expressive in ZQL.

What we *should* do instead:

- Centralise constants (`PAGE`, `INITIAL_COMMENT_LIMIT`, etc.) in
  `packages/zero-schema/src/consts.ts`.
- One small helper for the `+1` sentinel:

  ```ts
  export function paginate<T>(rows: readonly T[], limit: number) {
    return { visible: rows.slice(0, limit), hasMore: rows.length > limit };
  }
  ```

  Used at every list call site.
- Convention (in [`guidelines/conventions.md`](../guidelines/conventions.md)): every list query
  takes `{ limit: limit + 1, search?, ...filters }`.

### Frontend — *build one good list primitive*

This is where generalisation pays off. Sketch:

```tsx
<DataList
  rows={visible}
  isLoading={status?.type !== 'complete'}
  hasMore={hasMore}
  onLoadMore={() => setLimit(l => l + PAGE)}
  loadMode="button"            // | "infinite"
  renderRow={(row, i, selected) => <CustomerRow ... />}
  renderHeader={<ColumnHeader ... />}
  empty={<EmptyState ... />}
  keyboardNav={{ onSelect, onActivate }}   // wires j/k/Enter
/>
```

Replaces bespoke shells in `customers.index.tsx`, `inbox-list.tsx`, plus
the settings list pages. Locks in:

- One pagination UX (button vs infinite, configurable).
- One skeleton style.
- One keyboard model.
- One empty-state slot.

### Search params as state

Per the TanStack Router team's "Search Params Are State" post: filters,
sort, pagination, view preferences belong in the URL, validated at the
route level via `validateSearch` + Zod. This gives us:

- Deep linking and shareable filtered views.
- Back-button correctness.
- Type-safe reads via `useSearch`.
- Single source of truth (no local state duplicating URL state).

`customers.index.tsx` is a clean candidate to convert first
(`?search=foo&sort=name&limit=100`).
